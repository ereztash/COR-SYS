## 2. מקור Benchmark ו־48 שאלות (מה נמצא ב־repo)

- **48 שאלות:** מתועד ב־`docs/product-roadmap.md` (P2-3): 12 Macro + 15 Meso + 21 Micro. Tier 2 מופעל ב־≥7.0, Tier 3 ב־≥6.0.
- **Benchmark:** ב־roadmap (P3-4): "גבוה מ־72% מהארגונים בתעשיית ה־SaaS", מנתוני **simulation N=10,000**. ב־`src/lib/dsm-engine.ts` מוזכר: `integrated-model.md (N=10,000 simulation)`. הקובץ `integrated-model.md` לא נמצא ב־repo — לחפש ב־RAG או בתיקיית המחקר (למשל Claude / זהות עסקית).

## 3. מקרי עבר (Case studies) — המלצת CTO

**דעה:** כדאי מאוד להוסיף למחקר מקרי עבר שמחזקים את פרוטוקולי ההתערבות.

**למה:**
- **לגיטימציה:** "ארגון X היה ב־ND-3 ויצא ל־ND-1 אחרי BIA חיצונית ו־near-miss" — מחזק את ההמלצה.
- **חוויית מוצר:** בדו"ח / בעמוד התוצאות — "דומה למקרה Y שהצליח כך וכך" נותן תקווה ומוחשיות.
- **מכירה:** יועץ יכול להציג "זו אותה מתודולוגיה שעזרה ל־Z".

**איפה זה נוגע:**
- **מחקר:** מסמך (או RAG) עם 3–5 מקרים: פרופיל (DR/ND/UC), התערבות, תוצאה. אם יש נתונים — גם ציונים לפני/אחרי.
- **מוצר:** (אופציונלי) שדה "מקרה דומה" בפרוטוקול התערבות או ב־PDF — קישור למקרה קצר (אנונימי או עם שם).

**מאמץ:** מחקר — בינוני (איסוף + ניסוח). במוצר — קטן (טקסט/לינק בדו"ח).

## 4. לוגיקה עסקית — כניסה ו־Landing

**כניסה היום:**
- **מי נכנס:** ארגון (משתמש מקצה הארגון). בהמשך — יועץ.
- **איך:** **לינק** — הארגון נכנס דרך לינק, עם **הזדהות לפי גוגל** (Google Sign-In / OAuth).
- **לפני login:** מסך כניסה (Google); אין דף שיווקי ציבורי.

**Landing:**
- **מתי יהיה:** כשנצליח **למקם את זה כמוצר עובד**. עד אז — אין landing נפרד.
- **עד אז:** routing מינימלי — כניסה מאומתת (Google) לארגון; self-serve ולינקים בלי דף שיווקי.

ההחלטות האלה יקבעו: Auth = Supabase + Google provider; `src/proxy.ts` מפנה ל־login אם אין session; אין `/home` ציבורי כרגע.

## 5. עדכון Roadmap בהתאם

- **P2-3 (48 שאלות):** מאושר — להרחיב. מקור המבנה: מחקר + `docs/product-roadmap.md`.
- **P3-4 (Benchmark):** מקור הנתונים — חיפוש ב־RAG / integrated-model; ב־repo מופיע N=10,000 ו־percentile לפי סקטור.
- **P0-1 (Auth):** כניסה via **לינק + הזדהות גוגל** (Google Sign-In). Supabase תומך ב־OAuth עם Google.
- **P1-3 (Landing):** לא לממש כרגע — יהיה כשהמוצר ממוקם כמוצר עובד (סעיף 4).
- **P3-1 (Payment):** placeholder באפליקציה — לא מימוש מלא עד להחלטה.
- **מקרי עבר:** להוסיף כ־תנאי מחקר/תוכן לפני (או במקביל ל) הרחבת פרוטוקולי ההתערבות במוצר.
